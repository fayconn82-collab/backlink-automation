# 我是怎么用 Hermes + OpenCLI，把外链提交跑成半自动化 SOP 的

作者：小虾米

这篇文章是被公众号读者催出来的。

之前我在文章里提过，我最近一段时间一直在稳定跑一套自动化外链流程。每天给几个出海小站做白帽外链提交，记录平台、状态、避坑点，再定期追踪收录情况。

很多朋友问：

> 这套东西到底怎么搭？能不能写一篇能照着跑的？

所以这篇不是 SEO 理论文，也不是“外链多重要”的科普文。

这是一份实操手册。

目标只有一个：

> 让你照着步骤，把 Hermes Agent + OpenCLI + Chrome 这套半自动化外链流程跑起来，并且能正确判断哪些外链提交算成功，哪些不算。

本文配套附件会提供：

- 通用版 Skill
- 读者版外链平台列表
- Prompt 模板
- 工作目录模板
- 每日记录模板

这些附件不会直接写在正文里，避免正文太长。你可以下载后放到自己的工作目录里，让智能体直接读取。

这篇文章适合你，如果你正在做：

- 出海工具站
- 小型 SaaS
- AI 工具站
- 内容站
- 独立产品
- SEO 外链执行

它不适合你，如果你想要：

- 黑帽外链
- 群发垃圾外链
- 自动绕过验证码
- 自动注册大量账号
- 买链接渠道
- 保证排名暴涨

先说清楚，本文不会承诺“用了就排名暴涨”。

外链是长期工作，不是玄学速成。我们要做的，是把原来每天重复复制粘贴、填表、上传截图、记录状态的苦活，变成一套可以长期复用的 SOP。

说实话，独立开发者最缺的不是想法，是时间。

能少浪费一点，就少浪费一点。

## 1. 先看最终效果

跑通以后，你每天要得到的不是“我好像提交了几个外链”。

而是一张清清楚楚的计数表：

```markdown
| 网站 | 目标 | 已计数 | 还差 | 已计数平台 |
|---|---:|---:|---:|---|
| DemoSite | 3 | 3 | 0 | SaaSHub, SSL Labs, Host.io |
```

还有一张明细表：

```markdown
| 时间 | 网站 | 平台 | 提交/验证 URL | 状态 | 是否计数 | 备注 |
|---|---|---|---|---|---|---|
| 2026-xx-xx | DemoSite | SSL Labs | https://... | verified | 是 | 公开检测页包含域名 |
| 2026-xx-xx | DemoSite | Example Platform | - | paid | 否 | 最后一步要求付费，跳过 |
```

注意第二行。

它不是失败了就不记录。

恰恰相反，失败、付费、验证码、重复提交，这些都要记录。

因为外链自动化最重要的不是点按钮，而是判断：

- 哪些真的提交成功了
- 哪些只是填了表
- 哪些是草稿
- 哪些需要人工处理
- 哪些下次不要再浪费时间

没有记录，自动化只是帮你点鼠标。

有记录，才会变成自己的外链资产库。

## 2. 这套流程的整体原理

我现在用的是：

- Hermes Agent：负责理解任务、调用工具、判断状态、写记录
- OpenCLI：负责控制真实 Chrome 浏览器
- Chrome：负责复用真实登录态
- Markdown / CSV / Excel：负责记录结果

可以理解成：

```text
Hermes = 大脑
OpenCLI = 手和眼睛
Chrome = 真实操作现场
记录表 = 账本
```

![Hermes + OpenCLI 外链自动化流程图](article-assets/workflow.svg)

为什么不用纯脚本？

因为外链平台非常碎。

有的平台要 Google 登录，有的平台要 GitHub 登录，有的平台有验证码，有的平台点到最后才告诉你要付费，有的平台页面看起来填完了，其实 React 状态没保存。

纯脚本很容易卡死。

而 Hermes + OpenCLI 的优势是：

1. 能操作你已经登录好的真实 Chrome
2. 遇到验证码、登录、付费时能停下来让你接管
3. 能把失败原因写进记录
4. 能把平台经验沉淀到 Skill 里，下次复用

所以这不是 100% 全自动。

这是更现实的半自动化。

机器做重复劳动，人做关键判断。

## 4. 安装 Hermes Agent

项目地址：

<https://github.com/NousResearch/hermes-agent>

![Hermes Agent GitHub 仓库截图](article-assets/screenshots/github-hermes-agent-cropped.png)

安装命令：

```bash
curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
```

安装完成后检查：

```bash
hermes doctor
```

第一次使用需要配置模型：

```bash
hermes setup
```

或者进入模型选择：

```bash
hermes model
```

你需要准备一个可用的模型 API Key。

这里不展开讲每个模型供应商怎么申请。原则很简单：

> 只要 Hermes 能正常对话、能调用工具，就可以继续。

安装成功以后，可以进入 Hermes：

```bash
hermes
```

如果能看到 Hermes 的对话界面，说明第一步 OK。

【截图 2：Hermes 安装成功 / hermes doctor 成功截图】

![Hermes 版本示例](article-assets/hermes-version-example.svg)

常见问题：

### 4.1 hermes 命令找不到

可能是 PATH 没刷新。

可以重新打开终端，或者按安装脚本提示把 Hermes 加入 PATH。

### 4.2 模型调用失败

通常是 API Key 没配好。

检查：

```bash
hermes config path
hermes config env-path
```

确认配置文件和 `.env` 里有对应 Key。

## 5. 安装 OpenCLI

项目地址：

<https://github.com/jackwener/OpenCLI>

![OpenCLI GitHub 仓库截图](article-assets/screenshots/github-opencli-cropped.png)

OpenCLI 用来控制真实 Chrome。

先检查 Node 版本：

```bash
node --version
```

建议 Node.js 20+。

安装 OpenCLI：

```bash
npm install -g @jackwener/opencli
```

检查版本：

```bash
opencli --version
```

然后运行：

```bash
opencli doctor
```

如果 doctor 提示浏览器扩展没连接，需要安装 OpenCLI Browser Bridge Chrome 扩展。

扩展装好后，再运行：

```bash
opencli doctor
```

看到 daemon 和 browser extension 都正常连接，就说明 OpenCLI 可以控制你的真实 Chrome 了。

## 7. 准备网站资料

这是非常关键的一步。

智能体不是神仙。你不给资料，它就只能现场猜。

每个网站准备一个 Markdown 文件：

```bash
sites/demo-site.md
```

内容可以这样写：

```markdown
# DemoSite

- Website Name: DemoSite
- Website URL: https://example.com
- One-liner: A simple browser-based tool for resizing and compressing images.
- Short Description CN: 一个在线图片处理工具，可以在浏览器里完成图片缩放和压缩。
- Long Description EN:
  DemoSite is a lightweight browser-based image tool that helps users resize, compress, and prepare images without installing desktop software. It is designed for bloggers, indie hackers, marketers, and developers who need a quick image workflow.
- Target Users: indie hackers, bloggers, marketers, developers
- Categories: Image Tools, Productivity, Developer Tools
- Tags: image-resizer, image-compressor, browser-tool, productivity, web-tool
- Logo Path: assets/logos/demo-site-logo.png
- Screenshot Path: assets/screenshots/demo-site-home.png
- Contact Email: demo@example.com
- GitHub: https://github.com/example/demo-site
- Notes: Use this as a demo product. Do not expose private project data.
```

你自己的真实项目，把 DemoSite 换成自己的站点即可。

但如果你要公开截图，建议还是用 DemoSite。

必须准备的字段：

```text
[ ] Website Name
[ ] Website URL
[ ] One-liner
[ ] Long Description EN
[ ] Categories
[ ] Tags
[ ] Logo Path
[ ] Screenshot Path
[ ] Contact Email
```

Logo 和截图不是每个平台都需要，但很多产品目录会要求。

提前准备好，后面会省很多事。

## 8. 加载配套 Skill

这里是本文最重要的附件之一。

配套 Skill 文件：

```text
article-assets/backlink-automation-skill/SKILL.md
```

这个 Skill 是通用版，不包含我的真实站点、账号、API Key 或本地路径。

它包含的是我们这段时间沉淀下来的规则：

- 哪些平台类型适合提交
- 什么状态算成功
- 什么状态不算成功
- 遇到登录、验证码、付费怎么处理
- 每日记录怎么写
- 平台避坑点怎么判断
- 技术检测页怎么表达，不能包装成推荐

你可以用两种方式加载。

### 方式 A：放在项目目录里，让 Hermes 读取

把附件里的 `SKILL.md` 放到：

```text
backlink-workspace/skills/backlink-automation-submission/SKILL.md
```

然后对 Hermes 说：

```text
请先读取 skills/backlink-automation-submission/SKILL.md，
再根据 sites/demo-site.md 帮我跑一次外链提交流程。
如果遇到登录、验证码、邮箱验证、付费按钮或不确定的提交按钮，请停下来让我确认。
提交完成后，把结果记录到 records/daily/2026-xx-xx.md。
```

### 方式 B：安装到 Hermes 的 Skills 目录

```bash
mkdir -p ~/.hermes/skills/backlink-automation-submission
cp skills/backlink-automation-submission/SKILL.md ~/.hermes/skills/backlink-automation-submission/SKILL.md
```

然后在 Hermes 中说：

```text
请加载 backlink-automation-submission 这个 Skill，
然后根据 sites/demo-site.md 帮我提交一个外链平台。
```

如果你的 Hermes 支持 `/skill` 命令，也可以：

```text
/skill backlink-automation-submission
```

## 9. 第一次跑通：一个网站，一个平台

第一次不要贪多。

不要一上来就 5 个站 × 3 个平台。

第一轮目标只有一个：

> 让 Hermes 能读取网站资料，能通过 OpenCLI 打开浏览器，能填写一个平台，能记录结果。

你可以直接在 Hermes 里输入：

```text
请先读取 skills/backlink-automation-submission/SKILL.md。

现在我想为 sites/demo-site.md 里的 DemoSite 提交一个外链平台。

要求：
1. 优先选择无需登录或低阻塞的平台。
2. 使用 OpenCLI 控制真实 Chrome 打开平台页面。
3. 根据 sites/demo-site.md 填写网站名称、URL、描述、分类、标签等信息。
4. 如果遇到登录、验证码、邮箱验证、付费按钮、badge 要求或不确定的提交按钮，请停下来让我确认。
5. 提交后不要立刻算成功，先判断是否有 submitted、pending review、public listing、dashboard record 或明确 confirmation。
6. 把结果写入 records/daily/2026-xx-xx.md。
```

这段 Prompt 很长，但很重要。

因为它把边界说清楚了。

尤其是这句：

> 如果遇到登录、验证码、邮箱验证、付费按钮、badge 要求或不确定的提交按钮，请停下来让我确认。

一定要写。

不要让智能体乱点。

## 10. 提交过程中怎么人工接管

自动化不是说你完全不用看。

真正稳定的做法是：

- 机器打开页面
- 机器填写资料
- 机器上传素材
- 人处理关键判断

下面这些情况，必须人工接管：

```text
[ ] 账号登录
[ ] Google / GitHub OAuth
[ ] 输入密码
[ ] reCAPTCHA
[ ] 邮箱验证
[ ] 付费按钮
[ ] 需要选择付费套餐
[ ] 需要给自己网站加 badge
[ ] 账号安全提醒
[ ] 平台提示重复提交
```

比如遇到登录，你就自己在 Chrome 里登录。

登录完成后告诉 Hermes：

```text
我已经完成登录了，请继续当前平台流程。
```

如果遇到付费墙，就告诉它：

```text
这个平台最后一步需要付费，先跳过。
请记录为 paid / incomplete，不要计数，然后继续下一个平台。
```

如果遇到验证码：

```text
这个平台出现验证码，先记录为 captcha，不计数。
不要继续死磕，切换下一个候选平台。
```

这就是半自动化的核心。

机器做重复劳动，人做判断。

## 11. 什么才算成功？

这是全文最重要的一节。

很多人外链提交失败，不是不会填表，而是不知道什么算成功。

### 11.1 可以计数的状态

下面这些可以计数：

```text
submitted / pending review
public / live / verified
scheduled
HTTP 200 + 页面/最终 URL 包含目标域名
```

也就是说，至少要有一个明确证据：

- 平台说提交成功
- 平台说等待审核
- 生成了公开页面
- dashboard 里有记录
- 页面 HTTP 200，并且能看到你的域名

### 11.2 不能计数的状态

下面这些不能计数：

```text
requires login
captcha
email verification
paid
failed
not relevant
只填了表但没提交
只是创建了草稿
重复提交
没有明确回执
```

这部分一定要严格。

不严格，记录很快就会变成自嗨。

比如有个平台你填完资料，点击 Submit 后跳到 `$99 upgrade` 页面。

这不算成功。

应该记录为：

```text
paid / incomplete
```

再比如有个平台创建了 waiting-line 页面，但 `ready=false`、`launchDate=null`。

这也不算成功。

应该记录为：

```text
created / incomplete
```

## 12. 第二次跑通：一个网站，三个平台

第一次跑通以后，可以进入第二轮。

这次目标是：一个网站，三个不同类型的平台。

建议搭配：

```text
1 个产品目录
1 个技术检测页
1 个 Add URL / 搜索提交
```

比如：

```text
DemoSite：
- Launching Next（产品目录）
- SSL Labs（技术检测页）
- Active Search Results（Add URL）
```

这一步的目的不是凑数量，而是让你理解不同平台的判断规则。

### 12.1 产品目录

产品目录价值更高，但更容易遇到阻塞。

常见状态：

```text
submitted
pending review
public listing
paid upgrade
duplicate
requires login
```

### 12.2 技术检测页

技术检测页通常更稳定。

比如：

```text
SSL Labs
SecurityHeaders
Host.io
Wappalyzer
Shodan
VirusTotal
```

![Host.io 公开域名资料页示例](article-assets/screenshots/hostio-demo-public-page.png)

但一定要注意表达：

> 这是公开技术检测页，不是产品推荐页。

不要写成“某某平台推荐了我的产品”。

应该写成：

```text
公开 SSL/TLS 检测页，页面包含目标域名，作为技术资料页和索引发现补充。
```

### 12.3 Add URL

Add URL 类平台更像是搜索发现提交。

比如：

```text
Active Search Results
InfoTiger
Free Web Submission
```

它们不是高质量产品目录外链。

但如果有明确回执，也可以作为 index discovery 的补充。

## 13. 第三次跑通：多个网站，每站 3 个

等你单站三平台跑通后，就可以进入日常模式。

比如：

```text
今天目标：
- DemoSiteA：3 个
- DemoSiteB：3 个
- DemoSiteC：3 个

要求：
1. 每个网站 3 个有效提交/公开资料页。
2. 优先产品目录，其次技术检测页，最后 Add URL 补充。
3. 遇到登录、验证码、付费、badge 要求，不要死磕。
4. 每次成功或阻塞都写入 records/daily/YYYY-MM-DD.md。
5. 最后输出今日计数表。
```

这就是我们现在稳定跑的日常 SOP。

一个站一天 3 个，不多。

但持续做，效果会慢慢出来。

千万别一天猛发几十个。

白帽 SEO 不是爆破，是长期主义。

优点：

- 不一定需要登录
- 通常可以公开验证
- 适合做 public footprint
- 对新站 index discovery 有帮助

缺点：

- 不是产品推荐
- 不是强营销外链
- 检测结果可能显示问题
- 不能夸大价值

### 14.4 Add URL / 传统目录类

代表：

```text
Active Search Results
InfoTiger
Free Web Submission
```

原则：

> 能验证就记录，不能验证就不计数。

## 15. 常见平台坑点

下面这些是我们实战里踩过的。

### 15.1 Uneed

可能创建了 edit / waiting-line 记录，但：

```text
ready: false
launchDate: null
```

这种只是草稿，不算完成。

必须确认已经排期或有公开 listing。

### 15.2 SaaSHub

可能已有 managed listing。

如果已经存在，记录为：

```text
existing / active
```

不要重复提交，也不要当成当天新成果。

### 15.3 BetaList

最后可能出现付费套餐。

不付费就记录：

```text
paid / incomplete
```

不要误以为已经提交成功。

### 15.4 ToolFame

必须验证：

```text
https://toolfame.com/item/<slug>
```

或者 dashboard 中有记录。

它的编辑器可能看起来填了内容，但校验仍然是 0 字。

这种不算。

### 15.5 Launching Next

有明确的：

```text
Submission Received
```

才算成功。

如果进入 `$99` 升级页，跳过。

### 15.6 Fazier

免费发布可能要求：

```text
3 条 helpful comments
网站展示 Fazier badge
英文版
DR > 0
```

条件不满足，不计数。

### 15.7 Tiny Startups

可能会弹付费升级。

如果有免费继续按钮，可以选：

```text
No thanks, continue with free submission
```

但如果邮箱被判定为 auto-generated，需要人工处理。

### 15.8 ProductFame / Turbo0

这类动态表单容易出现：

- 看起来选了分类，但 React state 没保存
- 标签组件不接受自动输入
- 点击下一步后提示字段缺失

不要死磕。

记录为 incomplete，换平台。

## 17. 收录追踪怎么做

提交外链以后，还要分清三件事：

```text
页面存在 ≠ 搜索引擎收录 ≠ SEO 有效果
```

### 17.1 页面存在

检查：

- HTTP status
- final URL
- page title
- 页面是否包含目标域名

这只能证明页面存在。

### 17.2 搜索引擎收录

可以用 exact URL 查询。

比如 Yahoo Search：

```text
https://search.yahoo.com/search?p="提交后的完整URL"
```

因为 Google 直接抓取很容易遇到 bot 页面，Yahoo 可以作为 Bing 家族的参考。

但注意：

> 只有搜索结果里真实出现目标 URL，才算有搜索收录证据。

### 17.3 SEO 效果

真正的 SEO 效果要看：

- Google Search Console external links
- referring domains
- indexed pages
- impressions / clicks
- referral traffic

不要看到页面存在，就说“Google 已经收录”。

这是两回事。

## 18. 常见错误排查

### 18.1 Hermes 能启动，但模型不可用

检查 API Key。

```bash
hermes doctor
hermes config env-path
```

### 18.2 OpenCLI doctor 失败

常见原因：

- Chrome 扩展没装
- 扩展没启用
- daemon 没启动
- Chrome Profile 不对

先重新打开 Chrome，再跑：

```bash
opencli doctor
```

### 18.3 页面能打开，但不能点击

可能是页面动态加载、遮罩、弹窗或反爬。

让 Hermes 先读取页面状态，不要强点。

必要时人工接管。

### 18.4 表单填了，但提交失败

常见原因：

- 分类没保存
- 标签没保存
- 富文本编辑器内容没进 React state
- 图片尺寸不符合要求
- 邮箱被判定异常

处理方式：

```text
记录为 incomplete，不计数，换下一个平台。
```

### 18.5 平台要求付费

默认不付费。

记录为：

```text
paid / incomplete
```

除非你自己明确决定付费。

### 18.6 平台要求 badge

不要让智能体自动改你的网站代码。

记录为：

```text
badge required / not counted
```

如果你愿意加 badge，再单独处理。

## 19. 最后说几句

这套流程表面上是在自动发外链。

但我觉得真正有价值的，不是“今天多发了几个链接”。

真正有价值的是：

- 你有了标准化的网站资料
- 你有了可复用的平台库
- 你知道哪些平台能做，哪些平台是坑
- 你知道哪些状态算成功，哪些不算
- 你能把每天的外链工作记录下来
- 你能让智能体下次接着干，而不是每次从零开始

对于独立开发者来说，这很重要。

因为我们大部分人不是只有一个站。

今天一个工具站，明天一个小 SaaS，后天又想试一个新关键词。

如果每次都从零开始手工提交，真的很累。

但如果你把 Hermes + OpenCLI + Skill + 平台库 + 记录表这套东西跑通，新站上线后就可以复用。

说实话，这才是我最想分享的东西。

不是为了偷懒。

而是把有限的精力，从复制粘贴里解放出来，去做更重要的事情：

选词、做产品、看数据、改页面、继续上站。

Orz，独立开发者的时间真的太贵了。

能省一点是一点。
